# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

AniPortable is a Manifest V3 browser extension (popup-based) that lets users track/update their AniList anime & manga lists without leaving their current tab. It ships for both Chrome and Firefox from one codebase. Built with [Plasmo](https://www.plasmo.com/) + React + TypeScript + Apollo Client (GraphQL) + Tailwind.

## Commands

There is no `scripts` block in `package.json` — invoke the Plasmo CLI directly:

- `npm install` — install dependencies
- `npx plasmo dev` — start the dev build with watch/HMR (outputs to `build/chrome-mv3-dev`, load unpacked in `chrome://extensions`)
- `npx plasmo build` — production build (outputs to `build/chrome-mv3-prod`)
- `npx plasmo build --target=firefox-mv3` — Firefox build (outputs to `build/firefox-mv3-prod`, load via "Load Temporary Add-on" on `about:debugging#/runtime/this-firefox`)
- `npx plasmo package [--target=firefox-mv3]` — zip the production build for Chrome Web Store / AMO submission
- `npx tsc --noEmit` — typecheck only (there is no separate lint/test setup in this repo)

**Local setup requirement:** the AniList client ID comes from the env var `PLASMO_PUBLIC_ANILIST_CLIENT_ID`, inlined at build time by Plasmo. `Auth.login()` throws a descriptive error if the var is unset. AniList registers exactly one Redirect URL per app, and each build ends its OAuth flow at a different one, so each needs its own AniList app and its own gitignored env file: `.env.development` (`plasmo dev`), `.env.production` (`plasmo build`), `.env.firefox` (any `--target=firefox-*` build — it outranks `.env.production`). See README "Getting Started" for how to obtain each Redirect URL.

## Manifest / permissions

The MV3 manifest is **not** a standalone `manifest.json` — Plasmo generates it from the `"manifest"` key inside `package.json`. Edit permissions, `host_permissions`, background worker config, etc. there, not in `.plasmo/` (generated/gitignored) or `build/` (gitignored output).

Current permissions are intentionally minimal: `storage`, `identity`, and `host_permissions: ["https://graphql.anilist.co/"]` (exact-path match, no wildcard — the API is only ever called at the GraphQL root).

Plasmo rewrites that one manifest per target, so **don't hand-maintain Chrome/Firefox variants**:
- `background.service_worker` (what `package.json` declares) is emitted as-is for Chrome, and rewritten to `background.scripts` for `--target=firefox-*`. Firefox still doesn't support extension service workers — it uses non-persistent event pages — so this rewrite is what makes the Firefox build load at all.
- `browser_specific_settings.gecko` is kept for Firefox targets and **stripped** from the Chrome build. Only the `gecko` key survives — Plasmo drops **`gecko_android`**, so an Android-specific `strict_min_version` can't be set this way (verified; don't retry). That's why `web-ext lint` still reports one `KEY_FIREFOX_ANDROID_UNSUPPORTED_BY_MIN_VERSION` warning: `data_collection_permissions` needs Android 142 but desktop only needs 140. It's a warning, not an error, and this is a desktop popup extension — submission is unaffected.

If you change either, verify the generated `build/<target>/manifest.json` rather than assuming — Plasmo's target handling isn't obvious from its source.

## Architecture

**Two separate JS contexts, talking only through `chrome.runtime` messages/ports:**

1. **`background.ts`** — the MV3 service worker. Owns:
   - `Storage` — thin wrapper over `chrome.storage.local` (keys: `accessToken`, `user`, `pendingUpdates`).
   - `AniList` — raw `fetch`-based GraphQL client (used only from the background; separate from the Apollo client used in the popup). Includes `saveBulkMediaListEntries`, which dynamically builds one mutation with aliased `SaveMediaListEntry` calls per queued entry (`m<id>: SaveMediaListEntry(...)`).
   - **Debounced sync queue**: the popup never calls AniList mutations directly. It sends `{ action: "QUEUE_UPDATE", payload: { entryId, progress?, score?, status? } }`; the background merges it into an in-memory `pendingUpdates` Map, mirrors that Map to `chrome.storage.local` (`PENDING_UPDATES` key) on every change, and flushes via a single bulk mutation after a 5s debounce. The storage mirror exists because MV3 service workers can be killed while a `setTimeout` is pending — on worker restart, `restorePendingUpdates()` reloads and immediately re-flushes anything left over. A failed flush re-queues the batch instead of dropping it.
   - Flush is also triggered eagerly on logout and when the popup's `chrome.runtime.connect({ name: 'popup' })` port disconnects (i.e., popup closed).
   - `Auth` — OAuth via `chrome.identity.launchWebAuthFlow` (implicit grant, token parsed from the redirect URL fragment). Each login appends a random `state` param to the authorize URL — without it, Chrome can fail an immediate retry with "Authorization page could not be loaded" after the user closes/cancels the auth window, because it keys internal state off the exact URL. `login()` deliberately sends **no** `redirect_uri`: AniList redirects to whichever URL is registered on the app owning that `client_id`, and each browser only accepts its own (Chrome `*.chromiumapp.org`, Firefox `*.extensions.allizom.org`). That's why the client_id is per-build. Firefox's redirect URL is `https://<sha1(gecko.id)>.extensions.allizom.org/` — SHA-1 hex of the *declared* `browser_specific_settings.gecko.id`, **not** the random per-profile UUID Firefox also assigns each install (see `computeHash`/`getRedirectURL` in Gecko's `toolkit/components/extensions/child/ext-identity.js`). Because it depends only on the declared ID, one AniList app serves every Firefox user and both temporary and AMO installs. Changing `gecko.id` changes the redirect URL and invalidates the registered AniList app — re-derive the new one with `sha1(id)` (or read `chrome.identity.getRedirectURL()` from the background console) and update the AniList app to match. `broadcastAuthChange()` pushes `{ type: 'AUTH_CHANGED' }` to all tabs + the popup (if open) whenever login/logout/storage changes happen, so `useAuth` can resync without polling.
   - The file starts with `export {}` to force module scope — it has no imports, and without it TypeScript treats it as an ambient script, where `class Storage` collides with the DOM's global `Storage` type. Don't remove it.

2. **Popup** (`popup.tsx` and everything under `components/`, `contexts/`, `hooks/`) — a normal React app that is fully torn down and remounted every time the popup closes/reopens. Provider nesting in `popup.tsx`: `ApolloProvider` → `SettingsProvider` → `AniListDataProvider` → tab content.
   - `apollo/client.ts` — Apollo Client instance for the popup's own GraphQL reads/mutations (settings, viewer/list queries). Auth header is injected per-request by reading `accessToken` straight from `chrome.storage.local` (not from React state), so it always has the latest token.
   - `contexts/SettingsContext.tsx` — combines AniList-hosted user preferences (profile color, title language, adult-content flag, score format, row order — fetched via GraphQL and pushed back via `UpdateUser` mutations in `SettingsTab`) with purely local, extension-only settings persisted directly to `chrome.storage.local`: `manualCompletion`, `separateEntries`, `tabVisibility` (`'both'|'anime'|'manga'`, drives which of the Anime/Manga tabs `popup.tsx` renders), and `showAnimeStats`/`showMangaStats` (independent per-list toggles for the Stats tab). Changing `tabVisibility` resets both Stats toggles to match (hiding a tab hides its stats by default) but they stay independently overridable afterward — don't re-couple them. The context also surfaces `error` (from its own `VIEWER_QUERY`/`SETTINGS_QUERY`) alongside `loading`; every tab does the same for its own queries (see `lib/apolloErrors.ts` below) — a query that fails without checking `error` degrades silently into "0 items" instead of a visible failure, which is an easy trap to reintroduce when adding a new query.
   - `contexts/AniListDataContext.tsx` — a manual cache/dirty-flag layer *on top of* Apollo's own cache, per list (`anime`, `stats`, `manga`, `mangaStats`). This exists because Apollo's cache alone can't know that, e.g., completing an anime should invalidate the separately-fetched Stats query — components call `markXDirty()` after a mutation that affects another tab's data, and each tab's `useEffect` only refetches when `dirty` is true or the cache is empty. Since this state lives in React context (not storage), it only helps within a single popup session, not across popup close/reopen.
   - `hooks/useAuth.ts` — the only bridge from popup UI to the background's `Auth`/message handler; also subscribes to `AUTH_CHANGED` broadcasts.
   - `hooks/useStableOrder.ts` — freezes a sorted list's on-screen order while the user's mouse is over one of its cards (wired through `AnimeCard`'s `onHoverChange` prop, which piggybacks on the same `group-hover` that reveals its progress/score steppers), so editing a card's score/progress can't re-sort the list and shuffle a *different* card under the cursor mid-click. Item data still updates live — only position is held until the mouse leaves. `AnimeTab`/`MangaTab` call it unconditionally for every sub-list (Behind/Caught-Up, Reading/Completed, etc.) regardless of which is actually rendered, to satisfy rules of hooks.
   - `popup.tsx` builds its tab list from `TAB_DEFS`, filtered by `tabVisibility`, and tracks the active tab by `key` (not index) so a mid-session `tabVisibility` change can't leave `selectedKey` pointing at a now-hidden tab. The tab content wrapper is `flex-1 flex flex-col`, so its background always fills the popup's full height no matter how short the active tab's content is — every tab's loading/error/empty state (via `StateMessage`) depends on this.
   - `components/StateMessage.tsx` is the shared loading/error/empty-state UI (icon + text, centered, fills remaining height) used by all four tabs. Error messages are produced by `lib/apolloErrors.ts#getErrorMessage`, which inspects `error.networkError` rather than `error.graphQLErrors` — Apollo routes *any* HTTP status ≥300 through `networkError` (as a `ServerError`) even when the response body is valid GraphQL-shaped JSON, which AniList's own failure responses are. That's how 429 (rate limit), 403 (whole API disabled — the real message lives at `networkError.result.errors[0].message`, matched by substring since a 403 can also mean a real permission error), and 5xx are told apart from a plain GraphQL error.
   - `AnimeTab.tsx` and `MangaTab.tsx` are structurally parallel (same optimistic-update → `QUEUE_UPDATE` → dirty-flag pattern, same `useStableOrder` usage) and both render through the shared `AnimeCard` component (generic over "media entry with progress/score", used for both anime episodes and manga chapters).
   - Score formats (`POINT_100`, `POINT_10_DECIMAL`, `POINT_10`, `POINT_5`, `POINT_3`) and their max values are handled in `AnimeCard.getMaxScore`; keep new score-format logic there rather than re-deriving max values elsewhere.

## Known constraints

- Everything uses the `chrome.*` namespace, in both browsers — there is no `webextension-polyfill` and no `browser.*` usage. This works because Firefox aliases `chrome.*`, but with a catch: under **MV2** Firefox's `chrome.*` is callback-only, and only under **MV3** does it also return promises. The background (`Storage`) depends on the promise form (`await chrome.storage.local.get(...)`), while the popup uses the callback form — both are fine on MV3, but the promise form would silently resolve `undefined` and break auth on MV2. That's why the Firefox target must stay `firefox-mv3`, never `firefox-mv2` — the promise form would break silently under MV2, with a clean build.
- `strict_min_version` is `140.0`, set by the *later* of two constraints: MV3 `chrome.*` promise support landed in **128**, and `browser_specific_settings.gecko.data_collection_permissions` (required for AMO submission) landed in **140** — below that it's silently ignored, so users would install with no data-consent prompt. 140 is also the current ESR (128 EOL'd Sept 2025), so the floor costs no real users. If you lower it, `web-ext lint` will flag `KEY_FIREFOX_UNSUPPORTED_BY_MIN_VERSION`.
- `@apollo/client` is pinned to the exact version `3.8.10` via both `dependencies` and `resolutions`/`overrides` — a prior upgrade attempt caused build instability in this MV3 setup, so don't bump it without testing the actual built extension in Chrome, not just `tsc`/`plasmo build` succeeding.
- `tsconfig.json` uses `moduleResolution: "Bundler"` — if editing in VS Code and you see spurious `Cannot find name 'chrome'` errors, check the TS version in the status bar; VS Code's bundled TS can shadow the workspace's installed TypeScript and needs `TypeScript: Select TypeScript Version... → Use Workspace Version`.
- In `tailwind.config.js`, `white` is **not** pure white: `bg-white`/`text-white`/etc. resolve to `#edf1f5` (a light gray-blue — it's the popup's page background color), while `white-100` is the actual `#ffffff` (used for cards/panels that sit on top of the page background, e.g. `AnimeCard`, `CustomSelect`). Reaching for `bg-white` expecting pure white is a real trap here — check which one you actually want.
- `components/CustomCheckbox.tsx` mirrors its `checked` prop into local state only via `useState`'s initializer, so it needs the `useEffect(() => setIsChecked(checked), [checked])` sync it currently has to stay controllable from outside (e.g. `SettingsTab` programmatically flipping `showAnimeStats` when `tabVisibility` changes) — don't remove that effect as unnecessary/redundant-looking.
